# 팀장 업무 재설계 회의 플레이북 (ion-execs)

> 대상: ION 팀장(마케팅/세일즈/프로덕트/…) · 장소: 오프라인 90분 · 결과물: 팀별 plugin 설계 초안
> 전제 현실: 현재 `ion-execs` marketplace 에는 `exec-base` + `exec-marketing` **2개**만 존재하며, `exec-marketing` 만 ATMS MCP 기반으로 실제 동작한다. `exec-sales`, `exec-product` 등은 **아직 미구현**.

---

## 1. 회의 목적 (3줄)

- Claude Code plugin(`ion-execs`)을 팀장 개인 업무 인프라로 고정화해, 반복 브리핑·진단 업무의 리드타임을 줄인다.
- 마케팅팀장 파일럿(`exec-marketing` + ATMS MCP) 결과를 공유하고, **다음 파일럿 팀 1개**를 이 회의에서 결정한다.
- 팀별로 "자동화/준자동화/보조/비적용" 업무 매핑 초안과 agent·command 후보를 확정해 2주 내 구현 진입이 가능한 상태로 만든다.

---

## 2. 회의 전 준비 (각 팀장 지참물)

- 본인의 **1주일 반복 업무 리스트** 10개 내외 (빈도 · 소요시간 · 사용 툴 표기)
- 현재 사용 중인 **대시보드·소스 시스템** 목록 (ATMS, Notion, Slack, Gmail, Google Calendar, 자체 DB 등 — 현재 환경에 실제 붙어 있는 MCP: ATMS / Notion / Slack / Gmail / Google Calendar / Canva / Context7)
- **팀장 승인이 필요한 의사결정 유형** Top 5 (예: 광고 예산 증액, 채용 확정, 가격 변경)
- 최근 1개월 안에 **본인이 직접 숫자를 뽑았던 요청** 샘플 3건 (어디서 → 어떻게 가공 → 누구에게 전달했는지)
- 외부에 나가면 안 되는 **민감 지표/데이터** 목록 (타 팀에 공유 불가 포함)

> 가정: 모든 팀장은 Claude Code CLI 가 이미 설치돼 있고 `/plugin install exec-base@ion-execs` 를 마친 상태로 참석한다.

---

## 3. 회의 어젠다 (90분)

| 시간 | 세션 | 산출 |
|---|---|---|
| 0–10 | 배경 & ion-execs 구조 개요 (`exec-base` + 팀별 override 모델, `exec-marketing` 데모) | 공통 이해 |
| 10–20 | 마케팅 파일럿 라이브 시연: `/daily-brief`, `/diagnose-campaign`, `ads-ops` 에이전트 | 벤치마크 |
| 20–40 | 팀별 업무 리스트 낭독 → 4분류 워크시트 개별 작성 (섹션 4) | 팀별 매핑표 초안 |
| 40–55 | 조별 교차 리뷰 (마케팅팀장이 facilitator) | 오분류 보정 |
| 55–70 | plugin 설계 산출물 채우기 (섹션 5) — agent/command 후보 명세 | 팀별 초안 스펙 |
| 70–80 | 다음 파일럿 팀 선정 (섹션 6 스코어링) | 우선순위 결정 |
| 80–90 | 성공 지표·리스크·후속 액션 확정 (섹션 7–9) | 2주 체크리스트 |

## 4. 업무 재설계 프레임워크 (4분류 워크시트)

각 팀장은 지참한 반복 업무를 아래 4칸 중 하나로 매핑한다.

### A. 자동화 가능 — Claude + MCP 로 완전 대체
- **판정 기준**: ① 입력이 정형(기간·ID·채널) ② 출력이 정형(표·브리핑) ③ 판단 없이 조회/집계/포맷만 존재 ④ 오류 시 영향이 "다시 돌리면 끝"
- **마케팅 사례**: `/daily-brief` — 어제 성과를 `company_ad_performance_summary` + `company_roi_summary` 로 집계해 `exec-briefing` 포맷으로 출력. 팀장은 읽기만 한다.

### B. 준자동화 — Claude 초안 → 팀장 검토
- **판정 기준**: ① 데이터 수집·가설 생성은 도구로 가능 ② 최종 판단은 도메인 맥락·정책이 필요 ③ 잘못 나가면 되돌리기 비용 있음
- **마케팅 사례**: `/diagnose-campaign <id>` — `campaign_diagnosis` + `campaign_performance_summary` 로 원인 후보 2–3개와 권장 조치를 뽑지만, **집행 변경은 절대 자동 실행 금지**(에이전트 원칙). 팀장이 승인 Yes/No 를 찍는다.

### C. 보조 — 조회/정리만, 판단은 팀장
- **판정 기준**: ① 일회성·탐색적 질문 ② 출력 포맷이 매번 다름 ③ 결과를 그대로 쓰는 게 아니라 "재료"로 사용
- **마케팅 사례**: `ads-ops` 에이전트를 대화형으로 호출해 임의 캠페인을 파고들 때 (ad-hoc Q&A). 고정 커맨드화는 하지 않는다.

### D. 비적용 — plugin 대상 아님
- **판정 기준**: 대면 협상, 정성 평가, 1:1 피드백, 법적/인사 판단, MCP 로 데이터 접근이 불가능한 업무
- **마케팅 사례**: 대행사 미팅, 신규 매체 파트너십 협상 등은 plugin 화 대상에서 제외.

> 워크시트 규칙: 한 업무 = 한 줄. "현재 소요시간 / 자동화 후 예상 소요시간 / 근거 MCP tool" 3열을 반드시 채운다. 근거 tool 이 없는 A·B 항목은 C 로 강등.

## 5. plugin 설계 산출물 (회의 종료 시점까지 각 팀이 가져갈 것)

각 팀장은 아래 5종을 문서 1장으로 정리해 나간다. 마케팅(`exec-marketing`) 이 포맷 레퍼런스.

1. **Agent 명세 1–2개** (frontmatter: `name`, `description` + 본문 프로토콜)
   - 레퍼런스: `plugins/exec-marketing/agents/ads-ops.md` — 업무 프로토콜 4단계 + 원칙(수치 없는 추정 금지, 집행 자동 변경 금지).
2. **Slash command 후보 3개** (frontmatter: `description`, `argument-hint` + 본문 수행 절차)
   - 레퍼런스: `daily-brief.md`, `diagnose-campaign.md` — 하나는 일일 브리핑, 하나는 단건 진단, 하나는 주간/월간 롤업 형태 권장.
3. **필요한 MCP 서버 목록** + 필요 tool 네임 (없으면 "가정: 향후 붙일 것" 명시)
   - 가정: 세일즈팀은 CRM MCP(미구현), 프로덕트팀은 PostHog/Amplitude MCP(미구현) 필요 가능성.
4. **민감정보 경계선**: ① 이 plugin 이 접근 가능한 회사/워크스페이스 범위 ② 타 팀에 **절대** 노출되면 안 되는 지표 ③ 팀장 본인만 볼 수 있어야 하는 항목
5. **출력 포맷**: `exec-briefing` 을 상속할지, 팀별 커스텀 포맷을 쓸지 결정 (원칙상 공통 포맷 상속 권장)

> 산출물은 곧바로 `plugins/exec-<team>/` 디렉토리 골격으로 옮겨진다. `marketplace.json` 에 entry 추가, 초기 버전 `0.1.0`.

## 6. 우선순위 선정 규칙 (다음 파일럿 팀 1개)

3축 1–5점 스코어링, 합산 최고점이 다음 파일럿.

| 축 | 정의 | 1점 | 5점 |
|---|---|---|---|
| **반복 빈도** | 해당 업무가 주당 몇 회 반복되는가 | 월 1회 이하 | 매일 |
| **의사결정 비용** | 1건 처리에 팀장이 직접 쓰는 시간 + 지연 시 손실 | 10분 / 영향 미미 | 1시간+ / 매출·리스크 직결 |
| **MCP 연결 난이도 (역점수)** | 데이터 소스가 이미 MCP 로 붙어 있는가 | 신규 MCP 개발 필요 | 이미 붙어 있음 (ATMS, Notion, Slack, Gmail, GCal 등) |

- 동점 시: **되돌리기 비용이 낮은 팀** 우선(실패해도 회복 가능).
- 필수 탈락 조건: ① 데이터 소스가 오프라인·엑셀 수작업 전용 ② 팀장 본인도 프로세스가 고정화 안 돼 있음.

## 7. 성공 지표 (KPI, 파일럿 후 4주 측정)

- **팀장 주간 절감 시간**: 목표 ≥ 2시간/주 (해당 반복 업무 Before/After 기록)
- **브리핑 리드타임**: 요청 → 팀장 검토 가능 상태까지. 목표 ≤ 5분 (마케팅 파일럿 기준선)
- **사용 빈도**: plugin command 주간 호출 횟수 ≥ 5회/팀장
- **신뢰도**: 출력된 수치 중 출처(MCP tool명 + 기간) 누락 비율 0% — `exec-base` 규약 준수
- **의사결정 반영률**: "승인 필요 항목" 중 팀장이 실제로 당일 처리한 비율 ≥ 70%

> 측정은 팀장 자가 리포트 + 가능한 경우 `SessionEnd` 훅의 transcript 동기화(`log-sync.sh`) 를 활용. **가정**: `EXEC_LOG_TARGET` 환경변수로 로그 수집 경로를 지정하면 자동 수집 가능(현재 스크립트는 placeholder, 정책 미구현).

## 8. 리스크 · 거버넌스

- **민감 지표 노출**: 팀별 plugin 은 해당 팀 워크스페이스·회사 범위로 한정. 타 팀 지표를 조회하려는 프롬프트는 `exec-base/hooks/scope-check.sh` 에서 차단(현재 placeholder, 이번 회의 결정에 따라 룰 작성 필요). 차단 시 exit code `2` + stderr 로 사유 표시.
- **집행 변경 금지 원칙**: 광고 정지/증액, 캠페인 on/off, 예산 변경, 가격 변경, 메일 발송 등 **외부 상태를 바꾸는 작업은 plugin 에서 자동 실행 금지**. `ads-ops` 원칙("제안까지만") 을 전 팀 공통 규약으로 격상.
- **타 팀 데이터 접근 경계**: 같은 ATMS MCP 라도 세일즈팀이 마케팅 성과 원본에 접근하지 않도록 워크스페이스 단위 분리. 공유 필요 시 마케팅팀장이 요약본을 전달하는 플로우만 허용.
- **출력 신뢰도**: 수치 = 값 + 기간 + 출처(MCP tool명) 3종 세트 강제(`exec-base/CLAUDE.md`). agent/command 지시문에 반드시 반영.
- **로그/감사**: `SessionEnd` 훅은 실패해도 사용자 세션을 방해하지 않아야 함(exit code 무관). 감사 용도로 쓰려면 **가정**: 외부 저장소(S3/Notion DB)로 pipe 하는 스크립트를 추가해야 함 — 현재 미구현.
- **MCP 자격증명**: plugin 은 MCP 서버를 포함하지 않음. 팀장 개인 `settings.json` 에서 관리 — 팀장이 바뀌면 토큰 회수 프로세스 별도 필요.

## 9. 후속 액션 템플릿 (회의 직후 2주 체크리스트)

**Week 1**
- [ ] D+1: 각 팀장, 4분류 워크시트 최종본 Notion 에 업로드
- [ ] D+2: 다음 파일럿 팀, `plugins/exec-<team>/` 스캐폴드 PR(빈 agent/command + `plugin.json` + `marketplace.json` entry, `0.1.0`)
- [ ] D+3: 필요한 MCP 서버 목록 확정 → 미존재 MCP 는 담당자 지정
- [ ] D+5: agent 1개 초안 머지 (`ads-ops.md` 패턴 복제)

**Week 2**
- [ ] D+7: slash command 3개 중 1개 동작 확인 (본인 워크스페이스에서 실제 호출)
- [ ] D+10: `scope-check.sh` 에 해당 팀 범위 룰 추가
- [ ] D+12: 성공 지표(섹션 7) 측정 베이스라인 기록 (Before 값)
- [ ] D+14: 팀장 회고 15분 — 사용 빈도·신뢰도·절감 시간 보고, 다다음 팀 파일럿 Go/No-go

**담당 분배 원칙**: plugin 파일 편집은 해당 팀장(또는 지정 참모) 이 직접 커밋. 공통 베이스 수정이 필요하면 `exec-base` PR 로 분리.
